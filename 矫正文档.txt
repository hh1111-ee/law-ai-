2.3 技术背景
2.3.1 智能体设计思路与方法（当前实现与规划）
本项目为演示性 AI 法律咨询平台。仓库代码（如 `combined_server.py`）与 README 显示：后端以 `FastAPI` 为主，模型调用示例通过 Ollama HTTP API，数据以本地 Pickle 文件持久化，日志使用 Python `logging`。仓库中并未包含向量化检索、检索索引、RAG 流水线或多智能体调度的实现。

因此，文档中有关 RAG（检索增强生成）、多智能体（Multi‑Agent）、Coze 平台集成或 DeepSeek‑R1 等的陈述应视为“设计设想/未来扩展”，而非当前已实现功能。下文按“当前实现”与“未来可扩展”分别说明。

当前实现（仓库中存在或示例化的部分）
- 后端：`FastAPI` 提供 REST 与 WebSocket 接口，使用 `uvicorn` 部署。
- AI 模型调用：通过 Ollama HTTP API（本地/自托管示例），README 给出了调用说明。
- 数据：演示性地使用 Pickle 文件和 Excel/Pandas 加载案例数据以供展示与简单筛选。
- 日志与并发：使用 Python `logging`，示例包含并发控制的思路（如使用 `asyncio.Semaphore`）。

未来可扩展（文档中可改为规划）
- RAG：向量化、embeddding、检索索引与检索融合可作为后续实现；
- 多智能体：若需要可引入调度层与任务分发框架实现“需求分析/情感/法律咨询”分工；
- 平台/模型替换：Coze、DeepSeek‑R1 等可作为替代方案或训练辅助，但当前仓库未集成。
2.3.2 用户界面设计
遵循简洁、易用与专业的设计原则，用户界面以静态页面为主，包含友好引导、预设高频问题、智能补问（前端发起补充信息请求）与场景快捷指令，便于快速进入对应咨询流程。
2.3.3 数据来源与处理
1. 数据来源
本系统的核心数据均来自权威法律信息平台与官方渠道，确保物业法律知识、案例数据的专业性与准确性，具体来源包括：
裁判文书网、中华人民共和国最高人民法院官网发布的物业纠纷相关裁判案例；
北大法宝、无讼等专业法律平台的民法典（物权编、合同编）、《物业管理条例》、各地物业地方性法规等法律条文；
住建部门、房管部门发布的物业管理相关政策、管理细则与规范性文件。
2. 数据处理
针对获取的原始数据，开展全流程数据清洗与标准化处理，保障本地知识库的质量：首先对重复的法律条文、裁判案例进行去重处理；其次剔除无效、残缺、与物业纠纷无关的噪声数据；最后对所有数据进行格式化整理（如按纠纷类型分类、法条按效力等级标注、案例按裁判结果归类），构建结构化、可检索的物业纠纷专属本地知识库。
2.3.4 模型训练与调优（规划）
当前仓库未包含用于训练 DeepSeek‑R1 或类似私有大模型的代码。若未来需要提高情感识别、意图分类或多模块协作能力，可考虑：
- 收集并标注物业纠纷对话数据用于情感与意图模型训练；
- 使用可复现的微调流程与验证集评估模型性能；
- 在引入个性化或情感模块时，建立伦理审查与安全退路（例如检测自伤/危机时触发人工干预）。
2.3.5 系统部署与技术架构
1. 开发与部署核心技术栈
本系统采用前后端分离架构开发，后端以 Python3 为核心开发语言，前端基于原生 Web 技术实现，同时结合 AI 模型、实时通信、数据处理等技术组件，构建高可用、高响应的物业纠纷 AI 法律服务网站，核心技术栈与依赖如下：
技术层面	核心框架/工具	版本/核心用途
后端主框架	FastAPI	0.115.8，实现 REST API、WebSocket 实时通信，支持异步高并发
后端次要框架	Flask	3.0.3，兼容少量基础功能模块
ASGI 服务器	Uvicorn	0.34.0，部署 FastAPI 应用，支持 WS 协议
HTTP 客户端	Requests	2.32.3，调用 Ollama AI 模型 API、外部服务接口
数据处理	Pandas/Openpyxl	3.0.0/3.1.5，处理物业纠纷案例 Excel 数据、构建结构化 DataFrame
AI 模型集成	Ollama HTTP API	调用 gpt-oss-cloud 模型，结合 asyncio.Semaphore 控制并发
实时通信	WebSocket	FastAPI+asyncio 实现，支持用户私聊、咨询消息实时推送
数据持久化	Pickle	本地存储用户、帖子、消息等数据（users.pkl、posts.pkl 等）
日志运维	Python logging	基于 RotatingFileHandler 写入 combined_server.log，实现日志监控
前端核心技术	HTML+CSS+JavaScript	原生 DOM 开发，构建纯静态前端页面

前端网络请求	Fetch API	发起后端 REST API 调用（如 /search 检索、/api/legal 法律咨询）
前端第三方服务	OpenStreetMap/ip-api	实现定位、反向地理编码（调试中）
2. 开发与部署平台
开发环境：依托 Coze（扣子）新一代 AI 开发平台完成智能体的搭建、调试与优化，借助其强大的模型训练和工作流编排能力，快速实现物业纠纷场景智能体的功能开发；同时结合 Python 原生开发环境完成前后端代码的编写、测试与联调。
部署形式：系统以网站形式部署，前端为纯静态文件直接部署至 Web 服务器，后端服务通过 Uvicorn 启动运行，用户无需安装额外应用程序，通过浏览器即可访问物业纠纷 AI 法律服务网站，实现 7×24 小时在线咨询，大幅提升服务的便捷性与可及性。
2.3.6 核心智能体专项设计
1. 需求分析智能体
作为系统的 “入口调度中枢”，核心解决物业纠纷咨询中用户需求多元化的问题 —— 用户可能仅需专业物业法律解答、仅需情绪疏导，或两者均有需求。该智能体在用户输入文本后，通过意图识别模型完成需求类型判断与物业纠纷场景分类，并根据判断结果自动选择并调度对应智能体，实现 “千人千面” 的服务匹配。
为保障智能体高效运行，搭建了专属工作流与定制化提示词：工作流遵循 “输入解析→意图识别→场景分类→智能体调度→结果反馈” 的逻辑，实现标准化任务处理；提示词围绕物业纠纷场景定制，明确智能体的角色为 “物业纠纷咨询需求分析师”，确保解析与判断过程贴合物业法律服务的专业需求。
2. 情感陪伴智能体
针对物业纠纷易引发用户负面情绪的特点，打造物业纠纷场景专属情绪疗愈智能体，核心实现情绪识别、共情疏导、心理支持的功能，具体设计包括三大核心模块：
情感识别中枢：通过情绪识别与分析功能，实时测量用户情绪占比，结合物业纠纷场景的心理特点，运用专业心理技法分析情绪状态变化并给出针对性疏导建议，为用户提供即时情感陪伴，解决纠纷中的情感寄托诉求。
双轨应答系统
闲聊引擎：精准判断用户话题类型，针对物业纠纷相关的情绪倾诉，耐心倾听并共情回应；遇到玩笑型提问，以幽默风趣的语言互动；同时调用网络搜索整合最新物业相关民生资讯，结合时事热点调节用户情绪。
专业引擎：内嵌豆包大模型，针对用户因物业纠纷产生的心理困惑、诉求表达难题等复杂问题，给出专业、实用的建议。
安全与伦理：演示版本不包含生产级心理危机处理机制或个性化情绪干预。文档中关于按性别差异化应答的描述已移除，建议未来若要做个性化回复必须经过伦理与法律评审，并保留人工干预渠道。
3. 物业法律咨询智能体核心能力
作为网站的核心服务模块，该智能体基于Ollama 的 gpt-oss-cloud 模型 + 本地物业纠纷知识库 RAG 检索构建，具备三大核心能力：
快速检索与结构化解答：数秒内完成本地知识库检索，为用户输出结构化法律答复，包含核心问题解答、精准法条引用、相似裁判案例分析，答复结果可审计、可追溯；
并发限流与流式交互：结合 asyncio.Semaphore 实现 AI 模型调用的并发限流，保障系统稳定性；同时支持流式交互，让用户实时查看解答内容，提升咨询体验；
问题分级与人工移交：针对简单物业纠纷问题，自动给出标准化、可操作的维权建议；针对复杂、疑难的物业纠纷问题，系统自动标注并提供人工律师对接入口，实现 “自动化初筛 + 人工深度服务” 的衔接，显著降低人工初筛成本，提升客户响应速度。
2.3.7 系统模块化与可扩展性设计
代码采用管理器模式（用户管理器、消息管理器、帖子管理器等），实现基础对象抽象与文件持久化。当前为演示版本，需在生产化前完成：
- 将 Pickle 存储迁移到关系型或 NoSQL 数据库（MySQL/PostgreSQL/MongoDB）；
- 实施密码哈希、身份认证（如 JWT）、HTTPS 与输入校验；
- 增加审计、速率限制与监控；
- 若引入 RAG 或多智能体框架，补充向量数据库、检索层、调度层与可审计性设计。
